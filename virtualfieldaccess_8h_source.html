11// In code such as
12// pair z; write(z.x);
13// to evaluate the expression z.x, first z is pushed onto the stack, then as
14// it is not a record, instead of accessing its field in the usual way for a
15// record, a builtin function is called which pops z off the stack and
16// replaces it with z.x. virtualFieldAccess provides the access for the
17// virtualField 'x', and 'getter' is the access to the builtin function which
18// replaces z with z.x.
19//
20// If the virtual field z.x were mutable, then setter would access a builtin
21// function, which pops a real number and then z off of the stack, sets the
22// z.x to that new value, and puts the value back on the stack. In this case,
23// pairs are immutable, but other virtual fields may not be.
24//
25// Some virtual fields are functions, such as a.push for an array a. In rare
26// cases, code such as
27// void f(int) = a.push;
28// requires an object representing a.push, and so a getter for the field must
29// be provided. Most of the time, however, these fields are just called.
30// As an optimization, a caller access can be provided. If this access is
31// given, then a call to the virtualFieldAccess just translates into a call to
32// caller.
33class virtualFieldAccess : public access {
34 access *getter;
35 access *setter;
36 access *caller;

38public:
39 virtualFieldAccess(access *getter,
40 access *setter = 0,

42 : getter(getter), setter(setter) {}

58 return (bool)setter;
